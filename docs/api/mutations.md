# Mutations & RPCs

All writes stay on `@supabase/supabase-js`. Two flavours:

1. **Direct row writes** — `supabase.from(table).insert/update/delete/upsert(...)`.  Permitted by RLS only when the policy allows it for the caller.
2. **RPCs** — `supabase.rpc('name', { args })`.  These are `SECURITY DEFINER` Postgres functions that bypass RLS for the duration of the call so they can do multi-row work atomically. Used for everything that crosses a permission boundary (create user, ship order, attach school to event, etc.).

Every mutation in the dashboard is wrapped in a TanStack `useMutation` with an `onSettled` that invalidates the relevant cache keys.

## RPCs

All RPCs are `SECURITY DEFINER`. They check the caller's role inside the function body and raise an exception otherwise.

### Admin user / school management

| RPC | Args | Returns | Caller | What it does |
|---|---|---|---|---|
| `admin_list_school_leads` | — | `{ user_id, username, full_name, school_id, school_name, phone }[]` | admin | Lists every `school_lead` row from `auth.users` joined with their profile + school. |
| `create_school_with_lead` | `p_school_name, p_county, p_school_type, p_is_maker_space, p_contact_name, p_contact_phone, p_contact_email, p_lead_full_name, p_lead_username, p_lead_password` | `{ school_id, user_id, email, temporary_password }` | admin | Atomic: creates the `schools` row + the `auth.users` row + the matching `profiles` row + binds them. Returns the synthetic email + temp password so the admin can copy/share. |
| `admin_update_lead_credentials` | `p_user_id, p_full_name?, p_username?` | `void` | admin | Updates the profile's `full_name` and/or auth `email` for an existing lead. |
| `admin_reset_lead_password` | `p_user_id, p_new_password` | `void` | admin | Resets a lead's password (write to `auth.users.encrypted_password`). |
| `admin_delete_school` | `p_school_id` | `void` | admin | Deletes one school **and** every dependency in dependency order: product_units → orders → stock_ledger → event_attendances → event_schools → club_members → code_clubs → the auth user behind the lead → schools. |

### Activity attendance

| RPC | Args | Returns | Caller | What it does |
|---|---|---|---|---|
| `attach_school_to_event` | `p_event_id, p_school_id` | `void` | admin | UPSERTs into `event_schools` so the school is invited. |
| `mark_school_attended` | `p_event_id, p_school_id` | `void` | admin | Stamps `event_schools.attended_at` and inserts one `event_attendances` row per active student at the school. |
| `unmark_school_attended` | `p_event_id, p_school_id` | `void` | admin | Reverses the above: clears `attended_at`, deletes the per-student attendances for that event + school. |

### Orders pipeline

| RPC | Args | Returns | Caller | What it does |
|---|---|---|---|---|
| `create_consumable_assignment` | `p_school_id, p_product_id, p_quantity, p_notes?` | `void` | admin | Inserts an `orders` row with `fulfilled_by_school_id IS NULL` (ChipuRobo fulfils) + status='placed' + the recipient as placer. |
| `advance_consumable_to_production` | `p_order_id` | `void` | admin | Bumps status `placed → accepted → in_production`. |
| `ship_order` | `p_order_id` | `void` | maker_space (or admin) | Stamps `shipped_at`, status='shipped'. For durables sets `product_units.status = 'in_transit'`. |
| `mark_order_delivered` | `p_order_id` | `void` | school_lead (placer) | For consumables: writes `stock_ledger` `+quantity` and stamps `delivered_at`. For durables: stamps unit status `with_school`. |

### Certificates

| RPC | Args | Returns | Caller | What it does |
|---|---|---|---|---|
| `admin_bulk_issue_student_certificate` | `p_template_id, p_school_id, p_student_ids[], p_notes?` | `int` (count inserted) | admin | Inserts one `certificate_issuances` row per student; skips students that already have a non-revoked issuance for the same template. |

## Mutation patterns

### Optimistic update (TanStack)

Used where the response is deterministic — single-field toggles (active/inactive, in_club, attended).

```ts
const toggleActive = useMutation({
  mutationFn: async (vars) => { /* supabase.from('x').update(...) */ },
  onMutate: async (vars) => {
    await qc.cancelQueries({ queryKey: ['products'] });
    const prev = qc.getQueryData(['products']);
    qc.setQueryData(['products'], (rows) =>
      rows?.map((r) => (r.id === vars.id ? { ...r, ...vars.patch } : r))
    );
    return { prev };
  },
  onError: (_e, _v, ctx) => { if (ctx?.prev) qc.setQueryData(['products'], ctx.prev); },
  onSettled: () => qc.invalidateQueries({ queryKey: ['products'] }),
});
```

### Invalidate-on-settle (most cases)

```ts
const saveMutation = useMutation({
  mutationFn: async () => { /* supabase.from('x').insert(...) */ },
  onSettled: () => {
    void qc.invalidateQueries({ queryKey: ['x'] });
    void qc.invalidateQueries({ queryKey: ['leaderboard'] });
  },
});
```

### Realtime invalidation

For tables that drive cross-page state (orders, leaderboard inputs):

```ts
// src/lib/useOrderRealtime.ts and src/dashboard/Leaderboard.tsx
const channel = supabase
  .channel('feed-name')
  .on('postgres_changes',
      { event: '*', schema: 'public', table: 'orders' },
      () => qc.invalidateQueries({ queryKey: ['orders'] }))
  .subscribe();
return () => { void supabase.removeChannel(channel); };
```

RLS still applies to realtime broadcasts — the user only receives events for rows they could SELECT.

## Adding a new mutation

1. Write the SQL (CRUD or RPC) in a new `supabase/migrations/<timestamp>_<verb>_<noun>.sql`.
2. If it's an RPC, end the migration with `notify pgrst, 'reload schema';` so PostgREST + `supabase-js` pick it up without a restart.
3. In the page, wrap it in `useMutation`. Pick:
   - **Optimistic** if it's a one-field toggle and the result is predictable.
   - **Invalidate-on-settle** otherwise.
4. Invalidate every cache key the change touches. Cross-table mutations (e.g. `mark_order_delivered` changes orders + stock + units) invalidate `['orders']`, `['stock']`, `['units']`.
5. If multiple users could care about the change in real time, add a realtime channel listener (see pattern above).
